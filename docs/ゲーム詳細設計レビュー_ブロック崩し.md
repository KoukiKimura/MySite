## 1. レビュー観点

- 実装可能性
- 既存ルーター/ページ破棄規約との整合性
- テスト容易性
- プレイフィールに直結する衝突判定の妥当性
- 将来拡張時の変更容易性

---

## 2. 指摘事項と対応

| No | 指摘 | 対応内容 |
|----|------|----------|
| 1 | ステージデータを JSON にする必要性が低い | 初回は `stages.js` の静的定義とし、I/F だけリポジトリ化 |
| 2 | 高速移動時のブロックすり抜け懸念 | サブステップ更新を詳細設計に追加 |
| 3 | コンボ更新箇所が分散しやすい | `recordBlockDestroyed` と `recordPaddleBounce` に責務を集中 |
| 4 | 破棄漏れでイベントが残る可能性 | `pointer-controller` と `play-screen` に `destroy()` を必須化 |
| 5 | 一時停止中の状態再開が不明瞭 | `pauseSession` / `resumeSession` と `playScreen.pause()` / `resume()` を対応付け |

---

## 3. レビュー結論

- 初回実装としては、静的ステージデータ + 単一ボール + 単一耐久ブロックで妥当
- スコア、コンボ、残機はドメインで完結でき、Vitest で回帰確認しやすい
- プレゼンテーション層の責務は「入力・描画・衝突通知」に限定できる
- 実装後はソースレビューで次の点を重点確認する
  - ブロック衝突時の二重得点
  - ステージ遷移時のコンボ初期化
  - ルート離脱時のイベント解放
