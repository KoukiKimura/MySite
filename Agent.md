# ProjectTypingGame — プロジェクト構成ガイド

## プロジェクト概要

**ペンギンげーむず！** — ConoHa Wing上で複数のHTML5ゲームを提供するポータルサイト。  
DDD（ドメイン駆動設計）戦略設計を採用。

- **リポジトリ**: https://github.com/KoukiKimura/MySite.git
- **ホスティング**: ConoHa Wing

---

## ブランチ運用

```
main ── 本番公開用（リリース済みバージョン）
 └── develop ── 開発基盤（検証済みの安定版）
      └── future ── 開発作業用（新機能・変更はここで作業）
```

| ブランチ | 用途 | マージ先 |
|----------|------|----------|
| `main` | 本番公開バージョン | — |
| `develop` | 開発の基盤・統合ブランチ | → main（リリース時） |
| `future` | 日常の開発作業 | → develop（機能完成時） |

**運用ルール:**
- 日常の開発は `future` ブランチで行う
- 機能が安定したら `future` → `develop` へマージ
- リリース準備ができたら `develop` → `main` へマージ

---

## 前提・命名規約

### ファイル名は日本語とする
- ドキュメント・設計書等のファイル名は **日本語** で命名し、視覚的にわかりやすく管理する
- エクスプローラーやフォルダ一覧で内容が一目で把握できることを優先する
- ソースコード（`.js` `.css` `.html` 等の実装ファイル）は英語命名とする

**例:**
| 種別 | 命名例 |
|------|--------|
| 設計書 | `要件定義書.md`、`画面設計書.md`、`API仕様書.md` |
| 作業資料 | `調査メモ_認証方式.md`、`議事録_20260403.md` |
| 実装コード | `typing.core.js`、`main.css`（英語） |

## 設計方針

### DDD戦略設計

フロントエンド（`web/src/`）は以下の境界づけられたコンテキストで分割する：

```
web/src/
├── shared/         # 共有カーネル（Canvas基盤、共通UI、ルーティング）
├── portal/         # ポータルコンテキスト（ゲーム一覧、サイト情報、お問い合わせ）
├── games/          # ゲームコンテキスト（各ゲームの実装）
│   └── typing/     #   └─ タイピングゲーム
└── user/           # ユーザーコンテキスト（認証、ランキング）※今回はモック
```

各ゲームは `domain/` `application/` `infrastructure/` `presentation/` の4層構造を持つ：

| レイヤー | 責務 |
|----------|------|
| `domain/` | ビジネスルール、エンティティ、値オブジェクト、ドメインサービス |
| `application/` | ユースケース、アプリケーションサービス |
| `infrastructure/` | 外部依存（データ読み込み、リポジトリ実装） |
| `presentation/` | Canvas描画、UI、入力処理 |

## 設計書管理

### フォルダ構成ルール

```
docs/
├── 要件定義/         # プロジェクト全体・ゲーム別 要件定義書、ロードマップ
├── サイト設計/       # ポータルサイト（共通）の基本設計書・詳細設計書
└── ゲーム設計/       # ゲームごとの基本設計書・詳細設計書
```

### 命名規則

| ドキュメント種別 | 命名パターン | 例 |
|---------------|-----------|-----|
| 要件定義書（マスター） | `要件定義書.md` | — |
| サイト要件定義書 | `サイト要件定義書.md` | — |
| ゲーム要件定義書 | `ゲーム要件定義書_{ゲーム名}.md` | `ゲーム要件定義書_タイピング.md` |
| サイト基本設計書 | `サイト基本設計書.md` | — |
| サイト詳細設計書 | `サイト詳細設計書.md` | — |
| ゲーム基本設計書 | `ゲーム基本設計書_{ゲーム名}.md` | `ゲーム基本設計書_ブロック崩し.md` |
| ゲーム詳細設計書 | `ゲーム詳細設計書_{ゲーム名}.md` | `ゲーム詳細設計書_ブロック崩し.md` |

### 新規ゲーム追加時の手順

1. `docs/要件定義/ゲーム要件定義書_{ゲーム名}.md` を作成
2. `docs/ゲーム設計/ゲーム基本設計書_{ゲーム名}.md` を作成
3. `docs/ゲーム設計/ゲーム詳細設計書_{ゲーム名}.md` を作成
4. `docs/要件定義/要件定義書.md` の「サブドキュメント参照」「画面一覧」「DDD 6.3」テーブルに行を追加
5. `Agent.md` の「ドキュメント」テーブルに行を追加
6. `docs/要件定義/ロードマップ.md` に開発フェーズを追加

### バージョン管理方針

- 各設計書の先頭メタデータ表に「版数」「更新日」を記載する
- 軽微な修正は版数を上げずに更新日のみ変更する（例: 誤字修正）
- 構造的な変更（セクション追加・削除・大幅な仕様変更）は版数をインクリメントする
- `git commit` メッセージに変更内容を端的に記載する
